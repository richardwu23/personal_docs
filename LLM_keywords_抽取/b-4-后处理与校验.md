

可以。
在你这个场景里，**后处理与校验**不是一个“补丁层”，而是把 **LLM 的概率性输出** 变成 **检索系统可控、可解释、可评测的确定性中间表示** 的那一层。

结合你前面的前提：

* 数据规范化还不清楚
* 会先做数据质量探测
* 会先做最小 alias 表
* LLM query understanding 已经会封装成一个接口，内部用 prompt + LLM API
* 下游已经有 keyword search 和 semantic search

那么 `后处理与校验` 的本质职责就是：

**把 LLM 返回的 JSON，变成“可信、受约束、可执行”的 retrieval plan 输入。**

而且这层要独立存在，不能寄希望于 prompt 一次性把所有问题都解决。OpenAI 官方现在明确支持用 JSON Schema + `strict: true` 约束结构化输出，同时也建议把 schema 设计清楚、字段命名清晰，并用 evals 来验证结构设计是否真的适合你的用例；另外，prompt 指南也明确建议固定模型快照并用 evals 跟踪 prompt 行为。([OpenAI Platform][1])

---

## 1. 先说作用：这层到底解决什么问题

它主要解决 5 类问题。

### 1）防止“结构正确但语义错误”

LLM 用 Structured Outputs 时，能大幅降低“少字段、错类型、非法 JSON”这类问题，但它**不能保证语义一定对**。
比如它可以合法输出：

```json
{
  "normalized_terms": [
    {"source":"清爽","canonical":"ZnO","type":"ingredient","confidence":0.82}
  ]
}
```

这在 JSON 语法上可能完全合法，但在业务上是错的。

所以你必须再做一层：

* schema/字段合法
* 领域语义合法
* 类型归属合法
* 逻辑关系合法

这就是“后处理与校验”的第一职责。Structured Outputs 负责把输出约束到 schema，后续仍需要应用层自己 parse / validate。官方示例里也直接展示了收到 JSON 后再用 Pydantic/类型系统做二次解析与校验。([OpenAI Platform][1])

### 2）把开放世界输出收敛到受控词表

你的数据规范化还不明，LLM 很容易产出这些东西：

* 宽泛同义词
* 过度脑补的 ingredient
* 不在你索引里的 canonical term
* 词表里不存在的属性映射
* application / dosage form 的不稳定表达

而检索系统真正需要的是：

* 哪些词必须进 keyword must
* 哪些词只能进 should
* 哪些只适合 semantic query
* 哪些可以转成 filters
* 哪些必须丢弃

所以后处理层要做一个收敛：

**free-form understanding → controlled retrieval signals**

### 3）保护召回，不让 LLM “帮倒忙”

很多 query understanding 系统上线后，失败不是因为 LLM 完全不会抽，而是因为：

* 抽多了
* 扩太远了
* 把软约束当硬约束了
* 把猜测出来的词塞进 must
* 把不稳定属性硬转 filter

后处理层的核心价值之一，就是**限制 LLM 的破坏力**。

### 4）给 planner 提供稳定输入

planner 不应该直接吃原始 LLM 输出。
planner 应该吃的是已经过校验和裁剪的结构，比如：

* validated_intent
* accepted_terms
* rejected_terms
* filter_candidates
* expansion_terms
* ambiguity flags

这样 planner 的行为才稳定。

### 5）让系统可观察、可评测、可 debug

线上出了问题，你要能回答：

* 是 LLM 理解错了？
* 还是 normalization 错了？
* 还是 validation 过严？
* 还是 planner 把 should 误放进 must？
* 还是 keyword query 构建策略错了？

如果没有后处理层，错误都混在一起，根本没法定位。

## 2. 在整体链路里，它放在哪里

建议放在这里：

```text
User Query
 -> LLM Query Understanding
 -> Raw Structured Output
 -> Post-processing & Validation
 -> Validated Query Representation
 -> Retrieval Planner
 -> keyword_query / semantic_query / filters
 -> Search
```

注意这里有两个“输出”：

### A. Raw LLM Output

模型原始返回的结构化 JSON。

### B. Validated Query Representation

后处理层产出的、真正给 planner 用的对象。

生产里不要直接让 planner 消费 A，而应该消费 B。

## 3. 这层具体拆成哪些子阶段

我建议拆成 7 个子阶段。

---

### 阶段 1：结构校验

这是最外层。

检查：

* JSON 是否可解析
* required fields 是否齐全
* enum 是否合法
* list / object 类型是否正确
* `additionalProperties` 是否允许
* 字段长度是否超限
* 空字符串 / null / 重复结构是否存在

如果你已经用 Structured Outputs + `strict: true`，这一步的失败率会比自由文本低很多，但你仍然要在服务端自己做 parse + validate，因为应用代码最终还是要消费这个结构。官方文档也明确展示了 schema 定义、`strict: true`、以及响应回来后再做类型解析与 validation 的做法。([OpenAI Platform][1])

工程上建议：

* 用 JSON Schema 约束 API 返回
* 用 Pydantic / dataclass / msgspec 再 parse 一次
* 所有 parse 错误都转成统一错误码

例如：

```python
class ValidationErrorCode(str, Enum):
    INVALID_JSON = "INVALID_JSON"
    SCHEMA_VIOLATION = "SCHEMA_VIOLATION"
    UNKNOWN_ENUM = "UNKNOWN_ENUM"
    TOO_MANY_TERMS = "TOO_MANY_TERMS"
    INVALID_CANONICAL_TERM = "INVALID_CANONICAL_TERM"
    TYPE_MISMATCH = "TYPE_MISMATCH"
    CONFLICTING_FILTERS = "CONFLICTING_FILTERS"
```

---

### 阶段 2：字段标准化

这一层不是“语义规范化”，而是**结构整理**。

做的事情包括：

* trim 空白
* 全角半角统一
* 大小写统一
* 术语字符串 normalize
* list 去重
* 排序稳定化
* 空数组和 null 统一策略
* source term 和 canonical term 的最小清洗

例如：

* `" ZnO "` → `"zno"`（内部 key）
* `"防 晒乳"` → `"防晒乳"`
* `"TiO2"` / `"tio2"` / `"Titanium Dioxide"` 先转内部对齐形式，再做 alias 命中

这里最好形成两套值：

* `display_value`：保留原始或较可读形式
* `normalized_key`：内部比对用

---

### 阶段 3：alias / canonical 映射

这是你当前场景最关键的一层。

因为你已经说了：**数据规范化不明**。
那就不能假设 LLM 直接吐出来的 canonical term 就是你应该信任的 canonical term。

建议做成：

```text
raw source term
 -> lexical normalize
 -> alias lookup
 -> canonical candidate list
 -> choose canonical / unresolved
```

#### v1 建议规则

1. **先查人工 alias 表**

   * 高价值、小而准
   * ingredient / function / property / dosage form / application

2. **再查从语料盘点得到的候选映射**

   * 只做建议，不直接自动信任

3. **如果 alias 没命中**

   * 不要强制归一
   * 标记 `canonical_status = unresolved`

#### 结果最好不是简单字符串，而是：

```json
{
  "source": "物理防晒",
  "normalized_source": "物理防晒",
  "canonical": "inorganic_uv_filter_system",
  "type": "function",
  "match_method": "alias_exact",
  "canonical_status": "accepted"
}
```

或者：

```json
{
  "source": "不泛白",
  "normalized_source": "不泛白",
  "canonical": null,
  "type": "property",
  "match_method": "none",
  "canonical_status": "unresolved"
}
```

这里的原则是：

**宁可 unresolved，也不要 fake canonical。**

---

### 阶段 4：领域语义校验

这是最容易被忽略、但价值最大的部分。

即使 alias 映射到了 canonical，你还要判断它**在当前上下文里是否合理**。

比如：

#### 类型归属检查

* ingredient 不能被塞进 dosage_form
* dosage_form 不能被塞进 application
* property 不该直接作为 must ingredient

#### 上下文合理性检查

* “不泛白”更适合 property，不适合 ingredient
* “防水”更适合 property_target 或 claim，不适合 hard filter
* “敏感肌”可能更像 application / audience，不一定能直接下推为 filter

#### 业务约束检查

* 某些 canonical term 在你的索引字段里根本没有稳定承载字段
* 某些属性在当前索引版本只能走 semantic，不能走 keyword filter
* 某些 application / dosage form 只有在特定字段质量达标时才允许下推

这一层建议由**规则 + 配置**实现，不要交给 LLM 自己决定。

例如配置可以是：

```yaml
field_support_matrix:
  ingredient:
    keyword_fields: ["title", "abstract", "claims", "ingredient_norm"]
    filterable: false
  dosage_form:
    keyword_fields: ["title", "abstract", "dosage_form_norm"]
    filterable: true
  application:
    keyword_fields: ["title", "abstract", "application_norm"]
    filterable: true
  property:
    keyword_fields: ["title", "abstract"]
    filterable: false
```

---

### 阶段 5：逻辑一致性校验

这层看的是整个结构之间是否冲突。

例如：

#### 冲突类型

* `must_terms` 和 `exclude_terms` 有同一个 canonical
* `filters.application=["sunscreen"]`，但 `intent=lookup_hair_conditioner`
* `dosage_form=["spray"]` 和 `dosage_form=["stick"]` 同时被当强 filter
* `must_terms` 太多，已经严重压缩召回
* 一个 query 里同时出现多个互斥结构，但没有被标记 ambiguity

#### 约束强弱检查

* 高不确定性词不能进入 must
* unresolved 词不能进 hard filter
* 属性词默认不能直接做 must keyword clause，除非有特别规则

#### 召回保护检查

* must terms 数量上限
* filters 数量上限
* 每类 term 的上限
* 低支持字段的 filter 禁止下推

这一层其实是在做：

**检索可执行性验证**

不是所有“看起来合理”的理解结果都适合直接下游执行。

---

### 阶段 6：置信度裁剪与降权

LLM 给的 confidence 不必完全相信，但可以拿来做门控。

建议不要把 confidence 当概率真值，而是当**排序信号**。

一个非常实用的 v1 规则：

* `confidence >= 0.85` 且 alias 命中成功 → 可进 must 候选
* `0.65 <= confidence < 0.85` → 只能进 should / expansion
* `confidence < 0.65` → 不进 keyword，只能保留给 semantic rewrite 或直接丢弃
* unresolved term 即使 confidence 高，也不能进 hard filter

你还可以再叠加一个 `support_score`：

```text
final_accept_score =
  0.45 * llm_confidence
+ 0.35 * alias_match_score
+ 0.20 * field_support_score
```

这样比只看 LLM 自报 confidence 稳得多。

---

### 阶段 7：生成“校验后中间表示”

这一层的输出不要还是“原始理解结果”，而要改造成给 planner 用的结构。

例如：

```json
{
  "intent": "lookup_formulation",
  "language": "zh",
  "accepted_terms": [
    {
      "canonical": "zinc_oxide",
      "type": "ingredient",
      "strength": "should",
      "use_for": ["keyword", "semantic"]
    },
    {
      "canonical": "titanium_dioxide",
      "type": "ingredient",
      "strength": "should",
      "use_for": ["keyword", "semantic"]
    },
    {
      "canonical": "lotion",
      "type": "dosage_form",
      "strength": "must",
      "use_for": ["keyword", "filter", "semantic"]
    }
  ],
  "property_constraints": [
    {
      "canonical": "low_whitening",
      "strength": "soft",
      "use_for": ["semantic", "rerank"]
    },
    {
      "canonical": "non_greasy",
      "strength": "soft",
      "use_for": ["semantic", "rerank"]
    }
  ],
  "rejected_terms": [
    {
      "source": "敏感肌",
      "reason": "unsupported_filter_field"
    }
  ],
  "ambiguities": [],
  "needs_clarification": false
}
```

注意这里最关键的是 `use_for` 和 `strength`。
因为 planner 关心的不是“识别到了什么”，而是“这些东西能怎么用”。

## 5. 工程上怎么实现

我建议 v1 就按下面 5 个模块拆。

---

### 模块 A：Schema Validator

负责：

* JSON parse
* JSON Schema 校验
* Pydantic/msgspec 二次解析
* 默认值填充
* 基础长度限制

示意：

```python
def validate_schema(raw_text: str) -> QueryUnderstandingRaw:
    data = json.loads(raw_text)
    jsonschema_validate(instance=data, schema=QUERY_UNDERSTANDING_SCHEMA)
    return QueryUnderstandingRaw.model_validate(data)
```

---

### 模块 B：Normalizer

负责：

* 字符串 normalize
* alias lookup
* canonical 候选生成
* type 修正
* unresolved 标记

示意：

```python
def normalize_terms(raw: QueryUnderstandingRaw, lexicons: Lexicons) -> NormalizedQU:
    normalized = []
    for term in raw.normalized_terms:
        source = normalize_text(term.source)
        hit = lexicons.lookup(source, term.type)
        if hit:
            normalized.append(term.to_canonical(hit))
        else:
            normalized.append(term.to_unresolved())
    return NormalizedQU.from_raw(raw, normalized)
```

---

### 模块 C：Semantic Validator

负责：

* 类型归属校验
* 字段支持校验
* soft/hard 约束修正
* 互斥冲突检查
* filter 可下推性检查

示意：

```python
def semantic_validate(qu: NormalizedQU, support: FieldSupportConfig) -> ValidatedQU:
    accepted = []
    rejected = []
    for term in qu.terms:
        if not support.is_type_supported(term.type):
            rejected.append(reject(term, "unsupported_type"))
            continue
        if term.type == "property" and "filter" in term.use_for:
            term.use_for.remove("filter")
            term.use_for.append("rerank")
        accepted.append(term)
    return build_validated_qu(qu, accepted, rejected)
```

---

### 模块 D：Constraint Shaper

负责：

* must / should / exclude 再分配
* 置信度裁剪
* 数量上限控制
* expansion 限制
* unresolved terms 的降级

示意：

```python
def shape_constraints(vqu: ValidatedQU) -> PlannedInput:
    must_terms = []
    should_terms = []
    for term in vqu.accepted_terms:
        if term.canonical_status != "accepted":
            term.strength = "soft"
        if term.score >= 0.85 and term.type in {"dosage_form", "application"}:
            must_terms.append(term)
        else:
            should_terms.append(term)
    must_terms = cap_terms(must_terms, max_n=4)
    should_terms = cap_terms(should_terms, max_n=8)
    return PlannedInput(must_terms=must_terms, should_terms=should_terms)
```

---

### 模块 E：Fallback Manager

负责：

* schema fail
* semantic fail
* over-constrained fail
* null output
* timeout

对应降级：

1. 原 query 直接进 semantic
2. 原 query + 轻规则分词进 keyword
3. 禁用 filter，只保留 rewrite
4. 禁用 query expansion

---

## 6. 最重要的一点：后处理不要偷偷“修”，要显式记录

每一次修正都要可追踪。

建议日志里分开记录：

```json
{
  "raw_query": "我想找清爽不泛白的物理防晒乳配方",
  "prompt_version": "qu_v1",
  "model_snapshot": "gpt-4.1-2025-04-14",
  "raw_llm_output": {...},
  "schema_valid": true,
  "normalization_actions": [
    {"source":"防晒乳","action":"alias_map","result":"lotion"},
    {"source":"物理防晒","action":"canonical_rewrite","result":"inorganic_uv_filter_system"}
  ],
  "validation_actions": [
    {"term":"non_greasy","action":"demote_to_soft_constraint","reason":"property_not_filterable"},
    {"term":"zinc_oxide","action":"move_from_must_to_should","reason":"expansion_term"}
  ],
  "final_validated_output": {...},
  "fallback_used": false
}
```

这样你后面做误差分析时，能看到：

* LLM 原来想表达什么
* 系统修了什么
* 为什么修
* 修完后检索效果有没有提升

OpenAI 的 prompt 指南明确建议生产里固定模型快照，并建设 evals 去跟踪 prompt 和模型变化带来的行为变化；这对你这种“LLM 输出不是最终答案，而是中间表示”的模块尤其重要。([OpenAI Platform][2])

## 7. 如何和“数据质量探测 + 最小 alias 表”结合

你前面提到的两个前置动作，会直接决定后处理层怎么做。

### 情况 A：字段质量还不清楚

那后处理层就不能太激进地下 hard filter。

例如你虽然识别出：

* dosage_form = lotion
* application = sunscreen

但如果你还没确认索引里 `dosage_form_norm`、`application_norm` 的覆盖率和准确率，就不要直接 filter 下推。
可以先这样：

* keyword query 加短语匹配
* semantic query 保留描述
* rerank 强调该约束
* filter 延后

### 情况 B：alias 表只有最小一批

那后处理层要把 term 分成 3 类：

1. **alias_confirmed**
2. **llm_inferred_only**
3. **unresolved**

然后规定：

* confirmed：可进入 keyword / filter 候选
* inferred_only：只能 should / semantic
* unresolved：默认不下推，只保留日志

这是很务实的 v1。

## 8. 这层最常见的错误设计

### 错误 1：只做 schema validate，不做业务 validate

这会导致“看起来全合法，但检索效果很差”。

### 错误 2：一旦 LLM 给了 canonical 就照单全收

在数据不干净、词表不完整时，这很危险。

### 错误 3：把 property 直接做硬过滤

例如“清爽”“不泛白”“稳定性好”这类词，很多时候更适合：

* semantic query
* rerank features
* soft keyword should

而不是 filter。

### 错误 4：静默修正，不留痕迹

后面没法 debug。

### 错误 5：没有 over-constraint guard

LLM 一旦多抽几个 must，召回就直接塌。

## 9. 一个适合你当前阶段的 v1 规则集

我会建议你先这样定。

### 接受规则

* alias 命中 + 类型合法 + 字段支持明确 → accepted
* LLM 推断但 alias 未命中 → weak accepted 或 unresolved
* 类型不合法 / 字段不支持 → reject

### 强弱约束规则

* ingredient：默认 should，除非 query 明确点名
* dosage_form：可 must
* application：可 must 或 filter 候选
* property：默认 soft，不做 hard filter
* process：默认 should，除非 query 明确强调

### 召回保护规则

* must_terms <= 4
* should_terms <= 8
* filters <= 3
* unresolved_terms 不入 filter
* inferred expansions 不入 must

### 降级规则

* schema fail → fallback
* accepted_terms 太少 → 用原 query
* accepted_terms 太多 → 截断
* 互斥约束过多 → 保留 core intent，软化其余约束
